  ---
  1. Battery Life — No Power Cable Needed

  Good industrial BLE beacons (Kontakt.io, Minew, Teltonika) run on AA or coin cell batteries.

  - At normal broadcast interval: 1–3 years on a single battery set
  - For a 1-week event: completely irrelevant. You could buy beacons today, install them, and battery dies in 2026
  - No 220v, no cables, no electrician needed. This is one of BLE's best features — fully wireless, mount anywhere with a screw or zip tie

  With 25,000–30,000 people, the only power concern is your phone charging stations, not beacons.

  ---
  2. Budget Not a Problem

  Good. Then use 400–500 beacons and place them every 8–10 meters along all green paths. Dense grid = better accuracy, better crowd penetration. This changes everything —
  budget was my main reason for suggesting GPS hybrid. If beacons-only is what you want and money isn't the constraint, it becomes much more viable.

  ---
  3. Yes, the "Wrong Distance" Problem is Solvable

  This is the most important technical question. Here is exactly how:

  Problem: RSSI (signal strength) fluctuates — beacon 20m away reads stronger than one 5m away for a moment.

  Solutions (all standard, well-known):

  a) Lower TX power on each beacon
  Set transmission power to minimum (-20 dBm or -12 dBm). Beacon now only reaches 5–8m maximum. Phone physically cannot detect a beacon 20m away. Problem mostly eliminated.

  b) Dense placement + only strongest signal wins
  If beacons are every 8m at low power, phone sees maximum 2–3 beacons at any time. Simple "strongest = nearest" logic works well.

  c) RSSI smoothing in app
  Take 5 consecutive readings, drop highest and lowest, average the middle 3 (Kalman filter or simple moving average). Eliminates momentary spikes. This is ~50 lines of code.

  d) Hysteresis rule
  Don't switch to "new nearest beacon" unless it is consistently stronger for 2 full seconds. Prevents flickering between two adjacent beacons.

  All four combined = reliable nearest-beacon detection. This is solved engineering, not research.

  ---
  4. Crowd Absorption — Real, But One Fix Solves It

  Yes, it is real. Human bodies are ~60% water and absorb 2.4GHz radio (same frequency as BLE and WiFi). At 30,000 people, a beacon at chest height surrounded by a crowd of 50
  people becomes nearly invisible.

  The fix is simple: mount beacons at 2.5–3 meters height.

  Signal broadcasts downward over people's heads and reaches phones at waist/pocket level without passing through bodies. Standard practice at airports, stadiums, and
  exhibitions worldwide. Every beacon needs a post, wall bracket, or temporary stand. This is your main installation consideration — not the beacons themselves but what you
  mount them on outdoors.

⏺ What Must Be in the App

  ---
  1. Onboarding (First Launch)

  - Language selection — WUF is international, minimum Arabic, English, French, Spanish (UN languages)
  - Single large button: "Start Navigation" — blind users cannot browse menus
  - Bluetooth permission request with audio explanation of why it's needed
  - Quick audio orientation: "Welcome to WUF. This app will guide you by voice. Shake phone at any time to repeat last instruction."
  - No registration, no login — zero friction

  ---
  2. Destination Selection

  - Fully voice-driven — VoiceOver (iOS) / TalkBack (Android) compatible
  - List of destinations spoken aloud:
    - Main Entrance
    - Registration
    - Conference Hall A / B / C
    - Accessible Toilets
    - Prayer Room
    - First Aid
    - Exit / Transport Pickup
  - User selects by double-tap or voice command
  - Confirmation spoken back: "Navigating to Registration. Starting now."

  ---
  3. Core Navigation Engine

  Beacon detection layer:
  - Continuously scans for BLE beacons
  - RSSI smoothing (moving average over 5 readings)
  - Hysteresis: only switch to new nearest beacon if stronger for 2 seconds consistently
  - Each beacon has a unique ID mapped to a position and an instruction

  Instruction logic:
  - Every beacon ID maps to a plain-language instruction in the database
    - e.g. Beacon AF:3C → "Continue straight. Main entrance is 40 meters ahead."
    - e.g. Beacon BB:12 → "Turn right. Follow the path. Registration is 20 meters."
  - When user enters beacon range → instruction fires once
  - If user stays in range and doesn't move → repeat instruction after 15 seconds
  - If no beacon detected for 30 seconds → "Signal lost. Please stop and wait. Help is being notified."

  Offline-first:
  - Full beacon ID → instruction map downloaded on app open
  - No internet needed during navigation
  - Works in airplane mode (Bluetooth still works)

  ---
  4. Audio Output

  - Text-to-speech, not pre-recorded audio — so instructions can be updated without new app build
  - Spoken at medium pace, slightly slower than normal
  - Every instruction repeated if user shakes phone
  - Volume auto-set to maximum on navigation start with user confirmation
  - Do not interrupt — finish current instruction before playing next
  - Earphone detection: if earphones plugged in, also play through earphones

  ---
  5. Haptic Feedback (Critical)

  Many attendees will be in noisy outdoor environments. Haptics must carry meaning without audio:

  ├──────────────────────────────────────┼──────────────────────────┤

  └──────────────────────────────────────┴──────────────────────────┘

  ---
  6. Help & Emergency

  - Single button or gesture (hold volume-down 3 seconds) → calls or messages a venue accessibility coordinator
  - Alternatively: plays a loud tone so nearby staff can find the user
  - "Signal lost" state after 30 seconds of no beacon → auto-notifies coordinator if connected
  - Simple panic button visible at all times on screen (large, high contrast)

  ---
  7. Accessibility Standards Compliance

  The app itself must be accessible:
  - Full VoiceOver (iOS) and TalkBack (Android) support — every button labeled
  - High contrast mode — many visually impaired users have partial vision, not total blindness
  - Font size respects OS accessibility settings
  - No gestures that require precision (no swipe patterns, pinch, etc.)
  - Screen can stay always-on during navigation (blind users won't notice battery drain from screen but will notice if app pauses)

  ---
  8. Admin Panel (Separate Web App or Simple Dashboard)

  You need this to operate the event:
  - Map of all beacon IDs with their current instruction text
  - Edit instructions without releasing a new app version
  - See which beacons are online/offline (if using Kontakt.io dashboard this is built-in)
  - Push updated route to all app users if a path changes during the event
  - Log of how many users navigated which routes (anonymous, for reporting)

  - Maps shown on screen (blind users don't use them)
  - QR codes

  - Social features
  - Ratings or feedback during event (add after)
